# Digging Up a Decade-Old Engine Buried Under 2,000 Files

*Part 1 of the BO2 High-Round Crash Fix series. New to the series? Start with the [Introduction](00-introduction.md).*

---

The Black Ops 2 Zombies high-round crash is one of the most thoroughly documented and
least understood bugs in competitive gaming. Players have catalogued it obsessively since
2012. They know the symptoms — usually a hard freeze with audio continuing for 3-5
seconds, then a crash to desktop with `Exception Code 0xC0000005`. They know the
correlations — aggressive playstyles crash earlier, long sessions crash regardless of
round number, Origins crashes faster than Buried. They've developed workarounds: disable
graphic content, cap FPS at 60, kill in small batches, avoid trap spam.

What they haven't known — until now — is why. Specifically, which code paths cause the
leak, what the exact engine limits are, and what can actually be fixed versus what's
architecture. Treyarch stopped patching BO2 in 2014. The crashes are frozen in amber,
shipped forever to every copy of the game.

This is the story of excavating the crash from the source.

---

## Getting the Source

BO2's game logic is written in GSC — the scripting language Infinity Ward developed for
Quake III, which Treyarch adapted across every title from World at War through Black Ops 3.
GSC compiles to bytecode that ships inside `.ff` (fast-file) archives bundled with the game.
The VM that runs it is embedded in the game executable.

The bytecode is not encrypted. A decompiler called [gsc-tool](https://github.com/xensik/gsc-tool),
written by xensik, can read it back to readable GSC source. The header on every decompiled
file says:

```gsc
// T6 GSC SOURCE
// Generated by https://github.com/xensik/gsc-tool
```

Running gsc-tool over the full BO2 zombies archive produces a source tree of roughly
**2,086 files** and approximately 600,000 lines of GSC. Most of this is generated,
verbose, and repetitive — but all of it is readable.

```
ZM/Core/
  maps/mp/zombies/
    _zm.gsc               — Main zombies game loop (4,500+ lines)
    _zm_spawner.gsc       — Zombie AI spawn and lifecycle (3,800+ lines)
    _zm_weapons.gsc       — Weapon system, PaP, box (2,700+ lines)
    _zm_utility.gsc       — Shared utilities: movement, animation, sound (5,000+ lines)
    _zm_ai_faller.gsc     — Faller (drop-in) zombie type
    _zm_magicbox.gsc      — Mystery Box system
    _zm_score.gsc         — Score tracking and powerup drops
    // ... 100+ more
  gametypes_zm/
    _hud_util.gsc         — HUD creation helpers
    _weapons.gsc          — Base weapon management
  // ...
Maps/
  Tranzit Diner/          — Map-specific scripts per zone
  Tranzit Town/
  Origins/
  // ...
```

The structure is comprehensible once you know how to navigate it. Core game logic is
`_zm.gsc`. Zombie AI lifecycle is `_zm_spawner.gsc`. Weapons and PaP are `_zm_weapons.gsc`.
Shared utilities — movement helpers, entity spawn wrappers, sound players — are
`_zm_utility.gsc`.

## The Audit Approach

With 600,000 lines of auto-generated decompiled code, random reading is useless. We needed
a systematic entry point.

High-round crashes in gaming engines almost always come from one of three sources:
1. **Entity leaks** — allocate memory, never free it
2. **Integer/float overflow** — arithmetic that wraps around or breaks comparisons
3. **Infinite loops** — code that enters a state it can't exit

For an entity-limited engine like IW/T6, the highest-probability cause is entity leaks.
Every game object — zombie, player, weapon pickup, sound emitter, physics prop, trigger
volume — occupies one slot in a fixed-size entity pool. If anything allocates slots and
doesn't free them, the count climbs until there's nowhere to spawn a new zombie.

**Audit step 1:** Find every `spawn()` call in the core zombies files and trace whether
its corresponding `delete()` is guaranteed to execute.

**Audit step 2:** Find every counter and accumulator that increases monotonically with no
reset mechanism and estimate how long before it hits a 32-bit or float limit.

**Audit step 3:** Find every loop that has a conditional exit and look for cases where
the exit condition can never be satisfied.

## What We Found

The audit surfaced 25 distinct issues across the core files. They fell into six categories:

| Category | Code | Count | Severity |
|----------|------|-------|---------|
| Entity leaks | EL-01 through EL-04 | 4 | Critical (1), Medium (3) |
| Infinite loops | IL-01 through IL-03 | 3 | Critical (2), High (1) |
| Integer/float overflows | OF-01 through OF-03 | 3 | High |
| State accumulation | SA-01 through SA-09 | 9 | Medium/Low |
| Config string exhaustion | CS-01 | 1 | Medium |
| Unknown/low risk | — | 5 | Low |

The severity ratings come from answering two questions: how reliably does this trigger at
high rounds, and how catastrophically does it fail when it does?

Three findings came out at the top of both axes:

**EL-01 (`lerp()` entity leak):** A `script_origin` is spawned for every zombie window
attack and may be permanently abandoned if the zombie dies at the wrong moment. Because
it's a function-local variable there's no way to clean it up from outside the function.

**IL-01 (`has_attachment()` infinite loop):** A while-loop condition that can never
become false if the target attachment isn't the first one in the weapon's name string.
One call with a three-attachment weapon freezes the server permanently. Since it's
in the Pack-a-Punch upgrade path, this triggers when enough unique weapons have been
PaP'd in a long session.

**OF-01 (zombie health overflow):** The exponential health formula hits int32 maximum
(2,147,483,647) near round 163, producing a health value the engine interprets as
"effectively infinite." Zombies become unkillable. The failsafe system then gets
stuck recycling the unkillable zombies forever — the round never ends.

## The Toolchain

Before any testing, we needed a development loop. Every fix hypothesis needs to be
runnable in the actual game. Three tools made this possible:

**gsc-tool by xensik** — `gsc-tool -m comp -g t6 -s pc <file.gsc>` compiles GSC source
to T6 bytecode. The output (a `.gscbin` file) goes into Plutonium's custom script
directory. Running `map_restart` in the Plutonium console reloads all scripts without
restarting the game.

**Plutonium T6** — the actively maintained PC client for BO2. It loads custom compiled
scripts from `%LOCALAPPDATA%\Plutonium\storage\t6\scripts\zm\`. Every custom script with
an `init()` function gets that function called after base game init. This is where we
attach watchdogs, override function pointers, and thread monitors.

**`build.sh`** — a wrapper around gsc-tool that compiles all scripts in `scripts/` and
the patched game overrides, deposits compiled output in `compiled/t6/`, and reports
pass/fail per file:

```bash
compiling zm_highround_patch.gsc ... ok
compiling _zm_weapons.gsc (override) ... ok
compiling _zm_utility.gsc (override) ... ok

results: 11 passed, 0 failed
```

The limitation we discovered on day one: Plutonium T6 uses **compile-time,
namespace-scoped function resolution**. You cannot override a base game function by
defining something with the same name in a custom script. The compiled FF bytecode has
hardcoded references to the functions it compiled against — no external definition can
intercept them. This constraint shapes every architectural decision that follows.

## Why This Matters Now

BO2 released in 2012. These bugs have existed since launch. Why write this in 2026?

Three reasons:

**Plutonium is the active platform.** The official Steam client has had no updates since
2014 and is effectively dead. Plutonium T6, the community PC client, has an active player
base, a modding ecosystem, and the script-loading infrastructure that makes systematic
testing possible. Everything in this series works on Plutonium.

**MrTLexify's round 255 run.** In January 2024, MrTLexify completed the world's first
verified round 255 on Buried solo — a run lasting approximately 120 hours. He confirmed
that 256 is unreachable: the round counter is a uint8, it caps at 255. The run
demonstrated both the upper bound and that it *is* theoretically reachable with the right
map and strategy.

But 120 hours on Buried is not representative of other maps. Origins — widely considered
the best-designed zombies map — has a documented session ceiling of roughly 26 hours. The
difference is architectural, and it's fixable.

**We have the source.** gsc-tool gives us readable decompiled code from the shipped
bytecode. Any claim in this series about why something crashes is traceable to a specific
function and line number. We are not speculating about the crash — we are reading the
code that produces it.
