# Methodology

How we approached the audit of Black Ops 2's zombies codebase, what tools we used, and why we structured the analysis the way we did.

## Objective

Identify the root causes of crashes and soft-locks that occur during high-round play (round 100+) in Call of Duty: Black Ops 2 Zombies mode. The goal is not just to catalog bugs, but to build a causal model: which specific code patterns lead to which specific failure modes, and in what order.

## Source Material

### The Decompiled Scripts

The analysis is based on GSC (Game Script Code) scripts decompiled from Black Ops 2's compiled game files using [gsc-tool](https://github.com/xensik/gsc-tool) by xensik. Every file in the repository carries this header:

```
// T6 GSC SOURCE
// Generated by https://github.com/xensik/gsc-tool
```

This confirms they are machine-decompiled, not hand-written. The decompilation is high-fidelity — GSC decompiles cleanly because it's a relatively simple, high-level scripting language. Variable names are preserved from the original compiled bytecode, and control flow is reconstructed accurately. The main caveat is that some constructs may be slightly restructured by the decompiler (e.g., `for` vs `while` loops), but the logic is equivalent.

Map-specific variants (e.g., `_zm_magicbox_tomb.gsc` for Origins, `_zm_magicbox_prison.gsc` for Mob of the Dead) extend the core with additional functionality but share the same fundamental patterns and bugs.

## Analysis Approach

### What We Were Looking For

The analysis was structured around five categories of failure mode, prioritized by their likelihood of causing high-round crashes:

1. **Entity leaks** — Spawned game entities (`script_origin`, `script_model`, `trigger_radius`) that are never deleted, permanently consuming slots against the engine's hard entity limit (~1024). This was our primary hypothesis based on community reports that crashes correlate with box hits, grenade usage, and game duration.

2. **Infinite loops / soft-locks** — Code paths that can spin forever, either freezing the server (hard loop) or preventing round progression (soft-lock). These would manifest as either an instant freeze or a round that never ends.

3. **Integer/float overflow** — Values stored in 32-bit types that grow without bound over hundreds of rounds. GSC uses 32-bit signed integers (max 2,147,483,647) and 32-bit floats (precision degrades above ~16 million).

4. **State accumulation** — Arrays, counters, and data structures that grow monotonically across the game session without cleanup between rounds. Even if they don't overflow, they increase memory pressure and can trigger engine-level instability.

5. **Race conditions** — GSC uses cooperative multitasking (`waittill`/`notify`/`endon`). Concurrent threads sharing global state can produce timing-dependent bugs that are hard to reproduce but become more likely under high-round server load.

### How We Analyzed Each File

For each of the 7 core files, we performed a full read-through looking for:

- Every call to `spawn()` — tracing whether the spawned entity always reaches a corresponding `delete()` on all code paths, including thread termination via `endon`
- Every `endon()` — checking whether it could terminate a thread that has outstanding entity allocations
- Every loop — checking for correct increment/termination conditions
- Every counter or accumulator — checking whether it's ever reset, and what happens when it grows large
- Every `level.` global — checking for concurrent access patterns
- Every `waittill` — checking whether the expected notify is guaranteed to fire

This is a static analysis approach — we're reasoning about code behavior without running it. Phase 2 builds runtime instrumentation to validate these hypotheses — the diagnostic HUD (`zm_diagnostics.gsc`) and stress test tool (`zm_stress_test.gsc`) are now written and compiling, awaiting runtime testing on Plutonium.

### What We Didn't Analyze

- **Native engine code (the XEX):** The GSC scripting layer sits on top of the T6 engine's native code (C++, compiled for PowerPC on Xbox 360 / x86 on PC). Issues in the native layer (memory allocation, rendering, networking) can't be seen from GSC. Our analysis covers what's visible from script.
- **Client-side scripts (.csc):** We focused on server-side `.gsc` files because crashes in zombies are server-side events. Client scripts handle rendering and effects.
- **Multiplayer scripts:** The `MP/` tree shares some common code but has different game logic. Our analysis is zombies-specific.

## Tools Used

| Tool | Purpose |

## Assumptions and Unknowns

These are things we believe to be true based on community knowledge but have not yet verified at runtime. Phase 2 tools are built and will confirm or correct these once we run them on Plutonium.

| Assumption | Basis | Needs Verification | Tooling Ready |
|-----------|-------|-------------------|---------------|
| Entity limit is ~1024 | Community reports, consistent across CoD titles | Yes — entity headroom probe in `zm_diagnostics.gsc` | Yes |
| GSC uses 32-bit signed integers | Standard for the IW engine family | Yes — score overflow test via `zm_stress_test.gsc` | Yes |
| GSC uses 32-bit floats | Standard for the IW engine family | Yes — powerup increment monitoring in diagnostics | Yes |
| `spawn()` returns `undefined` when entity limit is hit | Assumed from engine behavior | Yes — entity fill command (`/st fill`) will test this | Yes |
| Entity leaks are permanent for the session | No evidence of engine-level GC for leaked entities | Yes — headroom min tracking across rounds | Yes |
| Decompiled source accurately represents compiled behavior | gsc-tool is well-tested for T6 | Low risk, but noting it | N/A |
