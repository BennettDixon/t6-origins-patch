# Toolchain Setup

Live log of setting up the development environment. Every command, error, version, and workaround is recorded here so the blog can tell the full story and readers can reproduce the setup.

## Prerequisites

- Plutonium T6 installed and working (BO2 zombies launches and plays) — **needed for runtime testing only, not compilation**
- gsc-tool by xensik ([github.com/xensik/gsc-tool](https://github.com/xensik/gsc-tool))
- A text editor / IDE (Cursor)

## gsc-tool Installation

### Platform

Development machine: macOS (Apple Silicon / arm64). gsc-tool is a cross-compiler — it produces T6 bytecode regardless of host platform. We compile on Mac, deploy the compiled scripts to the Windows machine running Plutonium.

### Download

Prebuilt binaries are available from GitHub releases. No need to build from source.

```

Release: https://github.com/xensik/gsc-tool/releases/tag/1.4.10
Asset: macos-arm64-release.tar.gz (5,862 KB)
```

Available platforms in this release:

- `linux-arm64-release.tar.gz`
- `macos-amd64-release.tar.gz`
- `macos-arm64-release.tar.gz`
- `windows-arm64-release.zip`
- `windows-x64-release.zip`
- `windows-x86-release.zip`

### Install Steps

```bash
# Download
curl -L -o /tmp/gsc-tool-macos-arm64.tar.gz \
  "https://github.com/xensik/gsc-tool/releases/download/1.4.10/macos-arm64-release.tar.gz"

# Extract into tools/ (gitignored)
mkdir -p tools
tar xzf /tmp/gsc-tool-macos-arm64.tar.gz -C tools/
chmod +x tools/gsc-tool

# Verify
./tools/gsc-tool --version
# Output: GSC Tool 1.4.10.333 created by xensik
```

The binary is ~17MB, self-contained, no dependencies.

### gsc-tool CLI Reference (for T6)

```
./tools/gsc-tool -m comp -g t6 -s pc <file.gsc>    # Compile .gsc to bytecode
./tools/gsc-tool -m decomp -g t6 -s pc <file.gsc>   # Decompile bytecode to .gsc
./tools/gsc-tool -m parse -g t6 -s pc <file.gsc>    # Parse only (syntax check)
```

Key flags:
- `-m comp` — compile mode
- `-g t6` — Black Ops 2 (T6 engine)
- `-s pc` — PC platform (use `xb2` for Xbox 360)
- `-w <path>` — working directory for `#include` resolution
- `-y` / `--dry` — dry run (don't write files)
- `-d` / `--dev` — enable dev blocks in output

## Compilation Workflow

### What gsc-tool Produces

Input: `scripts/zm_test.gsc` (source, 52 lines)
Output: `compiled/t6/zm_test.gsc` (bytecode, 1,018 bytes)

The compiled output has the same `.gsc` extension but contains binary bytecode. gsc-tool places output in a `compiled/<game>/` directory relative to the input.

### Include Resolution

Our scripts use `#include` directives like:
```gsc
#include maps\mp\gametypes_zm\_hud_util;
#include maps\mp\zombies\_zm_utility;
```

gsc-tool compiles these as **unresolved references** — it records the include paths but doesn't need the actual files at compile time. The game engine resolves them at load time against its own script cache. This means we can compile on Mac without having the full base game scripts available as includes.

**Verified:** Compilation succeeds without `-w` flag pointing to a scripts directory. The includes are recorded but not validated at compile time.

### Build Script

`build.sh` in the repo root:
- Finds all `.gsc` files in `scripts/`
- Compiles each one with gsc-tool
- Reports pass/fail counts
- Lists compiled output in `compiled/t6/`
- Has a commented-out deploy section for copying to Plutonium's script directory on Windows

```bash
$ ./build.sh

compiling zm_test.gsc ... ok

results: 1 passed, 0 failed

compiled scripts in: /Users/bennett/projects/t6-scripts/compiled/t6/
```

## Scripts

### Hello World (`zm_test.gsc`)

Created `scripts/zm_test.gsc` — a minimal script that:
1. Hooks into player connect/spawn events
2. Creates a HUD text element showing "t6-scripts loaded" in green
3. Displays for 5 seconds then destroys the HUD element
4. Compiles successfully to 1,018 bytes

This will be the first thing we test when we get to the Windows machine with Plutonium.

### Diagnostics HUD (`zm_diagnostics.gsc`)

A persistent HUD overlay showing live game internals. Compiles to 5,639 bytes.

Metrics displayed (updated every 0.5s):
- **Round state:** round number, zombie health, spawn queue, active AI count
- **Entity pressure:** live headroom (via spawn probe), all-time minimum headroom
- **Accumulation:** total kills, recycle count, playspace timeouts, box hits
- **Overflow risk:** powerup drop increment, combined score total, grenade attack counter
- **Warning:** red alert when entity headroom drops below 100

The entity headroom probe runs every 5 seconds: it spawns up to 200 `script_origin` entities in sequence, counts how many succeed before `spawn()` fails, then deletes them all. The count equals remaining entity slots.

A round-start console logger prints a one-liner with headroom, kills, and recycles at each new round.

### Stress Test (`zm_stress_test.gsc`)

A chat-command-driven testing tool for rapidly reproducing crash conditions. Compiles to 6,413 bytes.

Uses Plutonium's `level waittill("say", player, message)` to listen for commands typed with `say` in the game console. All commands prefixed with `/st`:

| Command | What it does |
|---------|--------------|
| `/st skip <N>` | Skip to round N (kills zombies, recalculates health/speed/spawn delay) |
| `/st kill` | Kill all active zombies instantly |
| `/st fill <N>` | Spawn N permanent entities to eat headroom (simulates leaks) |
| `/st drain` | Delete all entities spawned by fill |
| `/st score <N>` | Set player's score_total (for overflow testing) |
| `/st nade <N>` | Set player's grenade_multiattack_count |
| `/st box <N>` | Set level.chest_accessed |
| `/st dropinc <N>` | Set zombie_powerup_drop_increment |
| `/st health <N>` | Set zombie_health directly |
| `/st god` | Toggle invulnerability |
| `/st help` | Print all commands |

The round skip command replicates what `round_think()` does between rounds: kills all zombies, zeroes the spawn queue, sets the round number, recalculates zombie health using the same formula as `ai_calculate_health()`, adjusts move speed and spawn delay, then fires the `end_of_round` / `between_round_over` notifications so the game's round loop advances naturally.

The entity fill/drain pair lets us simulate a game that has been running for hundreds of rounds (and leaked hundreds of entities) without actually playing those rounds. Combined with the diagnostics HUD, this directly tests the entity ceiling crash hypothesis.

## Gotchas and Troubleshooting

### macOS Gatekeeper

On first run, macOS may block gsc-tool ("cannot be opened because it is from an unidentified developer"). Fix:
```bash
xattr -d com.apple.quarantine tools/gsc-tool
```
Or: System Settings > Privacy & Security > Allow anyway.

*(We didn't hit this — noting for blog readers who might.)*

### Cross-Platform Compilation

gsc-tool on Mac produces the same bytecode as gsc-tool on Windows. The `-s pc` flag targets the game's PC runtime, not the host machine. So compiling on Mac and deploying to Windows Plutonium is fully supported.

### Include Paths Use Backslashes

GSC `#include` paths use backslashes (`maps\mp\zombies\_zm_utility`), which is a Windows convention. This is fine — gsc-tool handles them correctly on all platforms. Don't convert to forward slashes.
